# 04 UI设计规范

## 目标
- 这份文档是当前 `aegissec` Web UI 的实现约束，用来防止后续新增页面或重构时偏离既有风格。
- 除非有明确产品需求变更，否则新增或修改界面时应优先遵守本规范，而不是重新发明一套视觉语言。

## 总原则
- 界面要简洁，去掉不必要的描述、说明、装饰性文案和重复元素。
- 中文优先；仅在技术名词必须保留英文时使用英文。
- 优先保留真实数据、真实状态、真实操作，弱化“解释 UI 本身”的文案。
- 同一页面只保留一个主视觉焦点，不要出现多个抢占注意力的大气泡或重复面板。
- 优先延续当前已落地的样式和交互，不要轻易改信息架构。

## 全局壳层
- 左上角固定品牌只显示 `AegisSec`，不要再附加 `A` 图标、标语或副标题。
- 全局顶部不再使用大号 topbar 气泡；页面内容从品牌线附近直接开始。
- 全局导航使用抽屉式左侧栏。
- 全局导航项只显示图标和名称，不显示解释性描述。
- 抽屉控制按钮使用 `<` / `>` 字符语义，不使用额外说明文案。
- 右侧主内容区域应尽量上移，与左上品牌所在水平线保持紧凑对齐，避免顶部大面积留白。

## 对话页
- 对话页是唯一允许保留独立会话抽屉的页面。
- 会话抽屉视觉上应与全局抽屉同一语言：紧贴左边界、窄宽度、抽屉式收起展开。
- 会话抽屉展开时显示 `新对话` 主按钮；收起时只显示 `+`。
- `新对话` 和 `+` 不能同时出现。
- 会话抽屉不显示额外说明文案。
- 对话主区域保持简洁：不要恢复顶部大气泡、额外说明区或重复状态框。
- 只允许消息区滚动，输入区固定在底部。

## 非对话页统一规则
- 除对话页外，工作台内部统一采用“单一主气泡容器”布局。
- 不要再回到左右分离的双面板结构。
- 页面内部可以有多个小 section/card，但它们都必须属于同一个外层主容器。
- 如果一个信息块不提供关键状态、关键数据或关键操作，应优先删除。

## Settings 页
- Settings 页只保留真实配置内容。
- 不再显示顶部状态总览、说明气泡、重复标题块或多余摘要。
- 当前结构应从 `配置` 表单直接开始。
- 不要重新添加 `当前状态` 大区块，除非后续出现明确的产品需求。

## Skills 页
- Skills 和 MCP 必须保持为两个独立页面，不要再合并成一个页面。
- Skills 列表使用固定尺寸卡片网格。
- 桌面端一行显示 2 到 3 个 Skills 卡片。
- Skills 卡片只显示：
  - 名称
  - 描述
- 描述过长时使用省略样式截断，不显示来源、作用域、路径、哈希、兼容性等次级信息。
- Skills 详情使用模态弹窗，不使用页面内右侧详情面板。
- Skills 弹窗只显示：
  - 名称
  - 描述
- Skills 弹窗必须相对于浏览器视口居中，不是相对于列表区域居中。

## MCP 页
- MCP 列表使用固定尺寸卡片网格。
- 桌面端目标是一行稳定显示 3 个 MCP 卡片。
- MCP 卡片高度要尽量紧凑，避免大面积空白。
- MCP 卡片只显示：
  - 名称
  - 连接开关
- 不显示传输方式、作用域、配置路径、能力统计等次级信息。
- MCP 不再使用详情面板或详情弹窗；它是一个直接操作的列表页。

## 图谱页
- `SessionGraphWorkbench` 不要随意跟随其他页面一起重构 UI。
- 若只是因为共享组件签名变化导致编译失败，只做最小兼容修复，不主动改它的视觉结构。

## 文案规范
- 避免这类文案：
  - 解释页面用途的长段落
  - 解释按钮本身用途的副标题
  - 重复页面标题的眉标、tag、kicker
  - 为了“显得完整”而添加的说明句
- 优先使用短标签：
  - `配置`
  - `技能列表`
  - `MCP 列表`
  - `新对话`
- 能由布局或图标表达的信息，不再额外写一段说明。

## 样式与实现约束
- 继续使用现有 plain CSS 体系，不引入新的 UI 框架。
- 优先在 `apps/web/src/styles.css` 的后部追加覆盖规则，避免大范围重写历史样式块。
- 新样式应优先复用现有语义类和视觉 token。
- 修改共享组件时，要先确认不会误伤对话页。
- 路由、数据获取、后端接口、页面信息架构如果不是本任务目标，不要顺手改动。

## 交互约束
- 抽屉、弹窗、按钮的交互路径必须唯一且清晰，不要保留重复入口。
- 视觉收起状态和展开状态必须一一对应，不能同时暴露两套控件。
- 能直接操作的列表项，不要再强行附带一个冗长详情区。
- 弹窗应优先保证稳定定位、可关闭、可重复打开，不要受页面滚动容器影响。

## 新增界面时的检查清单
- 这个页面是否出现了两个以上主气泡容器？如果是，先合并。
- 是否加入了并不承载关键数据/操作的说明文案？如果是，先删除。
- 是否把 Skills/MCP 又做回了详情面板？如果是，回退。
- 是否让对话页之外的页面回到了左右双面板？如果是，回退。
- 是否破坏了左上角固定 `AegisSec` 品牌和抽屉语言的一致性？如果是，回退。
- 是否在桌面端留下了明显多余的顶部空白？如果是，继续压缩。

## 变更流程
- 任何新的 UI 改动，都应先对照本规范自检。
- 如果产品需求确实要求突破本规范，应先更新本文档，再改代码。
- 改完后至少做：
  - 浏览器回归
  - `pnpm lint`
  - `pnpm build`
