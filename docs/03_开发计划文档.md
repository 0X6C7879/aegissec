# 开发计划文档
**目标**：给个人维护者一份可以直接执行的开发路线  
**建议周期**：8 周  
**开发模式**：单人优先、每周一个里程碑、先闭环后增强

---

## 0. 最高约束

后续所有开发任务都必须围绕以下四类核心场景收敛，而不是把项目做成泛化攻击平台：
- SRC 场景中的自动化众测与主流漏洞发现
- 典型 CVE、云安全与 AI 基础设施漏洞的可复现验证
- 多层网络与 OA 环境中的多步攻击路径规划、权限维持分析与证据串联
- 基础域渗透实验环境中的企业核心内网推演

路线图中的 Session、Runtime、Skill、MCP、Workflow、Graph 与 Report 能力，都应服务于**授权、受控、可留痕的验证闭环**。

---

## 1. 开发原则

### 1.1 先打通主链路
优先完成：
- Session
- Chat
- Runtime
- Skill 扫描
- MCP 导入
- Workflow
- 图谱
- 报告

### 1.2 不提前平台化
第一版不要上：
- 微服务
- Redis 队列
- 图数据库
- 多租户
- 自建市场

### 1.3 先保证兼容“能识别、能展示、能导入”
不要在 V1 追求：
- 所有 Skill 语义完全一致
- 所有 MCP server 的高级特性都支持
- 复杂 OAuth 全覆盖

### 1.4 场景优先，围绕验证闭环建设
每周产出都要回答两个问题：
- 是否更贴近四类核心场景中的真实研究/推演任务？
- 是否增强了计划—收集—验证—归因—报告闭环，而不是增加无约束动作能力？

## 5. 每周检查点

每周结束前都检查：

### 技术检查
- 是否仍是单体优先？
- 是否新增了不必要的基础设施？
- 是否还能本地一键启动？
- 是否引入了过重依赖？

### 产品检查
- 本周工作是否更贴近 SRC、CVE / 云 / AI 基础设施、多层网络 / OA、基础域渗透模拟等核心场景？
- 是否更容易完成一轮计划—收集—验证—归因—报告闭环？
- UI 是否比上周更完整？
- 文档是否同步更新？

## 6. 风险预案

### 风险 1：MCP 兼容太花时间
预案：
- 先支持 stdio + remote HTTP
- SSE 放到后面
- 先做导入和发现，不追高级认证全覆盖

### 风险 2：Skill 语义兼容差异太多
预案：
- V1 只做识别与展示
- 保留 raw_frontmatter
- 不承诺所有字段立即生效

### 风险 3：图谱实现拖慢进度
预案：
- 先只做 Task Graph
- 再做 Causal Graph
- Evidence Graph 放到 P1

### 风险 4：前端耗时过多
预案：
- 优先功能页，减少装饰
- 先列表 + 抽屉 + 图，再做美化

### 风险 5：目标漂移成泛化攻击平台
预案：
- 每周都对照四类核心场景检查 mission fit
- 功能评审时优先保留验证、建模、证据与报告相关能力
- 不新增脱离授权边界和产品定位的能力承诺

## 8. Definition of Done（总体）
当满足以下条件时，可认定 MVP 完成：

- 可以在本地启动 Web 与 API
- 可以创建会话并对话
- 可以启动 Kali 容器并执行命令
- 可以识别 Claude/OpenCode/agents Skill 目录
- 可以导入 Claude/OpenCode MCP 配置
- 可以展示 MCP capabilities
- 可以运行一条完整 Workflow
- 可以看到任务图与因果图
- 可以导出 Markdown / HTML 报告
- 可以围绕至少一个代表性验证/推演场景形成可复现的闭环输出

## 10. 结论

对个人开源项目来说，最重要的不是“功能列表看起来很大”，而是：

**8 周内把围绕核心场景的主闭环做通，并让别人按 README 跑起来。**

本开发计划就是围绕这个目标设计的。
