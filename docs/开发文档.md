## 2. 项目定位

### 2.1 目标

AegisSec Agent 要解决的问题是：

> 如何让一个通用智能体围绕渗透测试目标，自主完成计划、任务拆解、受控执行、证据整理、图谱沉淀和结果汇总。

它不是：

- 单一场景脚本
- 只会聊天的 LLM 包装层
- 只负责调工具的工具集合器
- 仅为某个活动短期定制的 Demo

它应该是：

- 一个**以渗透测试流程为中心**的 Agent
- 一个**本地可运行、Web 可操作**的研究平台
- 一个**兼容 Skills / MCP** 的通用 Agent 工作台

### 2.2 适用场景

系统应天然覆盖以下类型场景：

- Web / API 目标的安全评估
- SRC / 众测中的常见漏洞发现流程
- 典型漏洞验证与上下文扩展
- 云与容器环境的安全检查
- 多步任务推进类靶场或实验环境
- 基础内网、主机与服务侧探索
- 需要图谱、证据和任务链追踪的复杂测试过程

### 2.3 第一阶段约束

为保证个人项目可落地，第一阶段做如下收敛：

- 单体后端，不做微服务拆分
- SQLite 优先，不先上分布式组件
- 角色多样，但统一在一个进程内编排
- Skills / MCP 做兼容层，而不是做生态平台
- Web UI 保持简洁，以 Workspace 为中心

## 3. 设计参考总结

参考项目体现出两种值得吸收的路径：

### 3.1 认知闭环路径

优秀的 Agent 型项目，通常不会把系统做成“单轮 Prompt + 工具调用”。

更有效的做法是：

- 把认知过程拆成多个职责
- 让规划和执行分离
- 让失败能被总结并反馈给下一轮计划
- 用图结构表达任务和证据关系

这类设计说明：

- Agent 必须有阶段性状态
- 工具只是执行手段，不是决策核心
- 图结构比线性任务列表更适合复杂任务推进

### 3.2 产品化控制台路径

另一个优秀方向是：

- 明确区分角色、Skills、工具、MCP、知识、Agent
- 给出清楚的 Web UI 管理入口
- 大量能力通过配置和目录约定扩展
- 把“可用、可管、可看”作为产品能力

这类设计说明：

- 一个好用的 Agent 平台需要统一控制面
- 配置化、目录化扩展远比硬编码好维护
- Web UI 是用户理解系统状态的重要界面，不应只是聊天框

## 4. 总体架构

### 4.1 架构目标

设计目标是：

- 保持单机可部署
- 保持结构分层清晰
- 保证后续易于演进
- 将执行、规划、图谱、兼容层分离

### 4.2 总体结构

```text
Frontend (React)
    ↓
FastAPI API Layer
    ↓
Application Services
    ├─ Session Service
    ├─ Workflow Service
    ├─ Graph Service
    ├─ Skill Registry Service
    ├─ MCP Registry Service
    └─ Runtime Service
    ↓
Agent Core
    ├─ Coordinator
    ├─ Planner
    ├─ Executor
    └─ Reflector
    ↓
Execution Layer
    ├─ Kali Docker Executor
    ├─ Skills Compatibility Layer
    └─ MCP Client Layer
    ↓
Persistence Layer
    ├─ SQLite
    ├─ File Artifacts
    └─ Graph Snapshot Storage
```

### 4.3 分层说明

#### Frontend
负责展示、输入、控制和可视化。

#### API Layer
负责 REST / WebSocket 接口与认证。

#### Application Services
负责编排业务流程，不直接承担底层执行。

#### Agent Core
负责真正的认知闭环。

#### Execution Layer
负责调用执行器、Skill、MCP。

#### Persistence Layer
负责状态保存、历史回放、工件管理。

## 5.3 Agent Core 模块

## Coordinator

### 输入
- 用户目标
- Session 当前状态
- Task Graph 快照
- 最新证据摘要

### 输出
- 下一轮行动请求
- 调用 Planner / Executor / Reflector 的顺序
- 状态更新

### 核心逻辑
```python
while session.active:
    if need_plan:
        planner.plan(...)
    runnable_nodes = workflow.pick_runnable_nodes(...)
    for node in runnable_nodes:
        executor.run(node)
        reflector.review(node)
    if reflector.requires_replan:
        planner.replan(...)
```

## Planner

### 输入
- Goal
- ScenarioType
- Existing task graph
- Evidence summary

### 输出
- phase plan
- node definitions
- dependency edges
- approval checkpoints

### 关键要求
- 不能输出大段不可执行自然语言
- 必须输出结构化计划
- 必须可转换成 DAG

推荐数据结构：

```json
{
  "phase": "environment_discovery",
  "nodes": [
    {
      "title": "collect_entry_points",
      "type": "task",
      "depends_on": [],
      "priority": 90,
      "approval_required": false
    }
  ]
}
```

## Executor

### 输入
- TaskNode
- Runtime policy
- Selected capability (runtime / skill / mcp)

### 输出
- stdout
- stderr
- exit_code
- artifact list
- structured observations

### 关键要求
- 所有执行都要结构化记录
- 不允许把执行结果只保存在聊天文本里
- 每一步必须带 trace id

执行记录结构：

```text
ExecutionRecord
- id
- session_id
- task_node_id
- source_type         # runtime / skill / mcp
- source_name
- command_or_action
- input_json
- output_json
- status
- started_at
- ended_at
```

## Reflector

### 输入
- 当前节点执行结果
- 任务上下文
- 证据图局部快照

### 输出
- success / fail / inconclusive
- failure_reason
- hypothesis_updates
- replanning_suggestion

### 失败分类
- runtime_error
- timeout
- insufficient_context
- invalid_assumption
- blocked_by_policy
- no_signal

## 5.4 Runtime 模块

### 职责
- 启动和管理 Kali Docker 容器
- 在容器中执行命令
- 限制资源、超时与挂载
- 回传结果和工件

### 推荐实现
- `docker` Python SDK
- 每个 session 绑定一个运行实例，或者按策略复用
- 独立工作目录映射到 `artifacts/{session_id}/`

### RuntimePolicy

```text
RuntimePolicy
- allow_network
- allow_write
- max_execution_seconds
- memory_limit_mb
- cpu_limit
- require_approval_for_exec
```

### 关键实现细节

#### 执行过程
1. 创建 exec
2. 读取 stdout/stderr
3. 等待退出
4. 记录 exit code
5. 收集生成文件
6. 将元数据写入数据库

#### 安全控制
- 默认禁用宿主敏感挂载
- 默认限制网络策略
- 执行超时后强制中断
- 所有命令都进入审计日志

## 5.7 Graph 模块

### 图谱类型

#### Task Graph
表示任务和依赖。

#### Evidence Graph
表示资产、观察、证据、结论的关联。

#### Causal Graph
表示“什么导致了什么结论”，重点体现因果链。

### 为什么用三种图

- Task Graph 解决“下一步做什么”
- Evidence Graph 解决“看到了什么”
- Causal Graph 解决“为什么得出这个判断”

## 10. 实现细节建议

### 10.1 先做结构化数据，再做 Prompt 优化

不要一开始把时间都花在 Prompt 微调上。

先保证：
- 任务是结构化的
- 执行记录是结构化的
- 图谱是结构化的
- UI 能完整消费这些结构化数据

### 10.2 任何重要结果都不能只存在聊天文本里

必须落到：
- task_nodes
- execution_records
- graph_nodes / edges
- artifacts

否则后续无法回放，也无法稳定渲染图谱。

### 10.3 Skills / MCP 必须做成注册表，而不是临时解析

原因：
- 前端需要展示
- Agent 需要查询
- 需要启停状态
- 需要记录兼容性问题

### 10.4 UI 要把复杂度集中在 Workspace

不要平均分散到很多页面。

因为对这个产品而言，用户真正持续使用的只有：
- Workspace
- Skills
- MCP

其他页面都是管理补充。

### 10.5 先用 SQLite + 文件工件存储

这是最适合当前阶段的方案：
- 易调试
- 易打包
- 易迁移
- 足够承载 v0.x 阶段

---

## 11. 非目标

当前版本不做：

- 多租户 SaaS
- 分布式 Worker 调度
- 图数据库强依赖
- 自建 Skill 市场
- 自建 MCP 市场
- 复杂企业权限系统
- 过重的报表模块

---

## 12. 验收标准

### 核心闭环验收
- 用户能创建 session
- Agent 能生成计划并形成 Task Graph
- Executor 能在 Kali Runtime 中执行受控动作
- 执行结果能写入日志和工件
- Graph 面板能实时显示任务与证据更新
- Skills / MCP 能被识别并展示
- 审批节点能暂停并继续执行
- 最终结果可导出和复盘

### UI 验收
- 主路径 80% 操作在 Workspace 内完成
- Skills / MCP 页面不超过 2 次点击即可到关键动作
- 图谱与任务树状态一致
- 日志与执行记录可追溯到具体节点

---

## 13. 总结

AegisSec Agent 的下一版应当明确：

- 它是一个通用渗透智能体
- 比赛场景只是它覆盖的应用场景之一
- 它的竞争力来自清晰的认知闭环、简洁有效的 Web UI、结构化图谱与兼容层设计

下一阶段最重要的不是继续扩充概念，而是把以下四件事真正做扎实：

1. 任务图
2. 受控执行
3. Skills / MCP 兼容层
4. 简洁明了的 Workspace
